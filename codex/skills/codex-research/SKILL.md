# Codex Deep Research

実装前にコードベースを読み、事実ベースの `research.md` を作る。推測で設計せず、既存コード・テスト・設定・ドキュメントから判断する。

## 手順

1. 対象プロジェクトと依頼内容を特定する。
2. `rg` / `rg --files` / `git log` / `git diff` などで関連ファイルを読む。
3. 既存の `.local/issues/<番号>/research.md` があれば読み、矛盾しないように更新方針を決める。
4. 調査結果を `research.md` として出力する。issue 番号やタスク名が分かる場合は `.local/issues/<番号>/research.md` を優先する。
5. 書き終えたら、要点・未確認事項・次の計画作成に使う材料を報告する。

## research.md の構成

```markdown
# Research: [タスク名]

## 依頼内容
[ユーザー要求の要約]

## 関連ファイル
- `path/to/file` - 役割

## 分かったこと
- 既存実装の事実
- データフロー、状態管理、API、依存関係

## 落とし穴
- 変更時に壊れやすい箇所
- テストや手動確認が必要な観点

## 未確認事項
- 必要なら要確認として残す

## 次の計画に使うメモ
- 実装方針の候補
- 既存パターン
```

## 原則

- 変更は加えず、読むことを優先する。
- diff から読み取れない仕様は創作しない。
- コード参照は具体的なファイルパスで残す。
- 長くしすぎず、実装判断に効く情報を優先する。
