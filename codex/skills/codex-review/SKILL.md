# Codex Review

コードレビューまたは分析を行う。レビューではバグ・リスク・回帰・テスト不足を優先し、好みの指摘や大きな設計論は必要な場合だけ扱う。

## レビュー手順

1. 対象差分を確認する。
   - PR やブランチ指定があればその差分を見る。
   - 指定がなければ `git status`、`git diff`、必要なら `git diff --staged` を確認する。
2. 影響範囲の大きいファイルから読む。
3. 既存テスト・型・lint・境界条件を確認する。
4. 指摘は重要度順に、ファイルパスと行番号付きで提示する。

## 出力形式

レビュー依頼の場合:

```markdown
## Findings
- [severity] `path/to/file:line` 問題点と理由

## Questions
- 必要な確認事項

## Notes
- テスト未実行などの残リスク
```

分析や相談の場合:

- 事実と推測を分ける。
- 既存コードのパターンを優先する。
- 実装案はシンプルに、必要なファイル単位で示す。

## 原則

- 指摘は再現可能な根拠に基づける。
- 重大な問題がない場合は「重大な指摘なし」と明記する。
- テストしていない場合は未実行と書く。
- diff から読み取れない仕様は創作しない。
