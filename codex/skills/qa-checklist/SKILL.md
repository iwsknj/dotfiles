# qa-checklist

現在のブランチに含まれる変更（=PR対象の差分）から、**UIのQA検証項目**を日本語のMarkdownで生成し、リポジトリの `.local/issues/<issue番号>/qa-checklist.md` に出力する。
派生として **CS / PdM 向けの簡潔版** (`qa-simple.md`) も生成できる。

## 引数（任意）

呼び出し時に以下が指定されることがある。順不同・どれも省略可。

- **issue番号** (`#339` / `339` / `issue 339`)
- **比較ブランチ** (`vs develop` / `develop` / `master`)
- **モード**: `simple` / `csv` / `業務向け` 等 → 簡潔版 (`qa-simple.md`) を出力
- **出力先パス**: 明示されていればそれを優先

引数が無ければ、対話で issue 番号と比較ブランチを確認する。issue 番号が分からない場合は、ブランチ名（`feature/339-xxx` 等）から推測してユーザーに確認する。

## 動作手順

### 1. 文脈を集める（並列で実行）

- `git status` / `git rev-parse --abbrev-ref HEAD` で現ブランチを把握
- `git branch -a` で比較対象（`develop` / `master` / `main`）の存在を確認。リポジトリの慣習に合わせる（the-terminal.jp 系は `develop`、weightly 等は `main` か `master`）
- `git merge-base <比較ブランチ> HEAD` でマージベースを取り、`git diff <マージベース>...HEAD --stat` と `git log <マージベース>..HEAD --oneline` で全体像を掴む
- リポジトリルートに `.local/issues/` が既にあるかを確認。同じ issue の既存ファイル（`research.md` / `plan.md` 等）があれば必ず読み、矛盾しない内容にする

### 2. 差分を読み込む

- 影響範囲の大きさに応じて `git diff <マージベース>...HEAD -- <パス>` を分割して読む
- フロントエンド（画面・URL・コンポーネント）／バックエンド（API・SQL・バッチ）／設定（マイグレ・環境変数）に分類しながら把握する
- 削除された機能・変更された画面・追加された画面を**洗い出す**。これがそのまま検証項目になる
- 「ゾンビ状態」（ローカルストレージ・Cookie・古いURLパラメータ等の残骸が残った状態）が起きうるかを必ず検討する

### 3. 検証項目を構造化する

通常版（QA担当・エンジニア向け）は以下のセクションを基本構成とする。**該当なしのセクションは省略してよい**。

| 章 | 内容 |
| --- | --- |
| サマリ | 何を変えたか・ねらい（ユーザー視点で1〜2段落） |
| 対象画面 / URL | 画面名・URL を表で列挙 |
| A. 正常系 | 各画面で必ず通る基本動作 |
| B. エッジケース / 境界値 | 0件・大量・権限なし等 |
| C. ゾンビ状態の確認 | 古いURL・LocalStorage・キャッシュ等で壊れないこと |
| D. リグレッション | 削っていない機能が今までどおり動くこと |
| E. 開発者向け追加確認 | API レスポンス構造・SQL本数・lint/test 等 |
| F. リリース後モニタリング | レイテンシ・5xx 率など |

各項目の書き方は **以下2スタイルのいずれか**。同じファイル内では混在させてよい。

- **テーブル形式**（手数の多いチェック向き）
  ```markdown
  | # | 検証内容 | 確認方法 |
  |---|---------|---------|
  | 1 | 一覧が変更前と同じ件数で表示される | 画面表示 |
  ```
- **見出し+操作/確認形式**（説明が必要なケース向き）
  ```markdown
  ### A-1. ページ表示
  **操作**
  対象タブを開く。
  **確認したいこと**
  - 一覧が正しく表示される
  - 画面エラーが出ない
  ```

### 4. 簡潔版（`qa-simple.md`）を作るとき

CS / PdM / ビジネス側向け。以下の構成にする。

- **何の対応？** — 1〜2段落でユーザー視点の言葉で（技術用語は最小化）
- **確認場所** — 画面名（URLは添えてよい）
- **所要時間** — 目安（10〜15 分など）
- **確認項目** — 番号付き見出し＋箇条書き。専門用語を平易な言葉に置き換える
- **不具合が見つかったら** — 報告フォーマット（手順 / ブラウザ / スクショ / 時刻）
- **おまけ：体感できる変化** — 改修前 / 改修後の対比表（任意）
- 末尾に詳細版（`qa-checklist.md`）への参照リンク

### 5. 出力する

- 出力先: `<repo-root>/.local/issues/<issue番号>/qa-checklist.md`（簡潔版は同じディレクトリに `qa-simple.md`）
- ディレクトリが無ければ `mkdir -p` で作る
- ユーザーが明示的にパスを指定したらそれを優先
- 同名ファイルが既に存在する場合は **必ず読んでから上書き or 追記の方針をユーザーに確認**（既存の検証実施ログを上書きしない）

### 6. 提示する

- 出力先の絶対パスを伝える
- 主要なセクション構成（章立てと項目数）を 5 行以内で要約
- ゾンビ状態・後方互換などで**判断に迷った点**があれば一言添える
- 簡潔版を作るかどうか未指定なら「CS/PdM 向けの簡潔版も作る？」と最後に聞く

## やらないこと

- 検証項目の**実施**（あくまで項目の作成のみ）
- ステージ・コミット・プッシュ
- `gh issue` / `gh pr` の操作（別スキルの責務）
- diff から読み取れない仕様の**創作**。不明な点は項目に「**要確認**」マークを付けて残し、本文で言及する
- 既存の `qa-checklist.md` を確認なしに上書きすること

## 参考: 実例の所在

過去に作ったQAチェックリストはリポジトリの `.local/issues/<番号>/` 配下にある（例: `the-terminal.jp/.local/issues/277/qa-checklist.md`、`/304/qa_checklist.md`、`/339/qa-simple.md`）。スタイルに迷ったらそれらを参照する。
