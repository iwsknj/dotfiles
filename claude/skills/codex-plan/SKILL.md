# Codex Plan Creation

Codex CLIを使用してコードベースを分析し、詳細な実装計画を`plan.md`に作成するスキル。調査フェーズ（codex-research）完了後に使用し、実際のコードベースに基づいた具体的な実装計画を生成する。

## 実行コマンド

codex exec --full-auto --sandbox read-only --cd <project_directory> "<request>"

## パラメータ

| パラメータ            | 説明                                       |
| --------------------- | ------------------------------------------ |
| `--full-auto`         | 完全自動モードで実行                       |
| `--sandbox read-only` | 読み取り専用サンドボックス（安全な分析用） |
| `--cd <dir>`          | 対象プロジェクトのディレクトリ             |
| `"<request>"`         | 依頼内容（日本語可）                       |

## 計画ドキュメントの要件

生成される`plan.md`には以下が**必ず含まれる**必要があります：

1. **アプローチの詳細な説明** - 実装方法の全体像と設計思想
2. **実際の変更を示すコードスニペット** - 具体的なコード例（既存コードの修正箇所も含む）
3. **変更されるファイルパス** - 影響を受けるファイルの一覧
4. **考慮事項とトレードオフ** - 実装時の注意点、代替案、パフォーマンスへの影響など

## 使用例

### 基本パターン: 新機能の実装計画

```bash
codex exec --full-auto --sandbox read-only --cd /path/to/project "システムを拡張して<ビジネス成果>を実行する新しい機能<名前と説明>を構築したいと考えています。これを実装する方法を概説した詳細なplan.mdドキュメントを作成します。コードスニペットを含めます。確認や質問は不要です。具体的な計画まで自主的に出力してください。"
```

### パターン1: ページネーション機能の追加

```bash
codex exec --full-auto --sandbox read-only --cd /path/to/project "リストエンドポイントは、オフセットではなくカーソルベースのページネーションをサポートする必要があります。これを実現する方法については、詳細なplan.mdを記述してください。変更を提案する前にソースファイルを読み取り、実際のコードベースに基づいて計画を立ててください。確認や質問は不要です。具体的なコード例まで自主的に出力してください。"
```

### パターン2: 参照実装を共有する場合

優れた実装が見つかったオープンソースリポジトリのコードを参照として共有する：

```bash
codex exec --full-auto --sandbox read-only --cd /path/to/project "並べ替え可能なIDを追加したいと考えています。以下のコードは、これを適切に実行するプロジェクトからのID生成コードです：

[参照コードをここに貼り付け]

同様のアプローチを採用する方法を説明するplan.mdを作成します。実際のコードベースを読み取り、具体的な実装計画を提案してください。確認や質問は不要です。コードスニペットを含む詳細な計画まで自主的に出力してください。"
```

### パターン3: 調査フェーズ結果を参照する場合

`codex-research`を実行済みで、既存の`research.md`がある場合：

```bash
codex exec --full-auto --sandbox read-only --cd /path/to/project "システムを拡張して新しい機能を構築したいと考えています。タスク番号/タスク名のディレクトリ配下に既存のresearch.mdがある場合は、その内容を参照して計画を立ててください。これを実装する方法を概説した詳細なplan.mdドキュメントを作成します。コードスニペットを含めます。確認や質問は不要です。具体的な計画まで自主的に出力してください。"
```

## 重要な原則

### 1. コードベースの事前読み込み

- **必ず**変更を提案する前にソースファイルを読み取る
- 実際のコードベースに基づいて計画を立てる
- 既存のアーキテクチャやパターンに従う

### 2. マークダウンファイルの使用

- Claude Codeの組み込みプランモードは使用しない
- マークダウンファイル（`plan.md`）を使用することで、完全な制御が可能
- エディターで編集したり、インラインノートを追加したりできる
- プロジェクト内で実際の成果物として保持される

### 3. 参照実装の活用

- オープンソースリポジトリで優れた実装が見つかった場合、そのコードを参照として共有する
- ゼロから設計するのではなく、具体的なリファレンス実装を使用して作業する
- これにより、より実用的で実証済みのアプローチを採用できる

### 4. 調査フェーズ結果の参照

- `codex-research`を実行した場合、タスク番号やタスク名のディレクトリ配下に`research.md`が生成される
- **必ず**計画作成前に既存の`research.md`を確認し、その内容を参照する
- 調査結果に基づいて計画を立てることで、より正確で実用的な計画を生成できる

## 実行手順

1. ユーザーから依頼内容を受け取る（新機能の説明、変更要件など）
2. 対象プロジェクトのディレクトリを特定する
3. **既存の`research.md`を確認する**（`codex-research`を実行済みの場合、タスク番号やタスク名のディレクトリ配下の`research.md`を読み取り、調査結果を参照する）
4. 必要に応じて参照実装のコードを依頼内容に含める
5. 上記コマンド形式でCodexを実行
6. 結果をユーザーに報告し、タスク番号 or タスク名のディレクトリ名配下に`plan.md`というファイル名で出力する

## 出力例の構造

生成される`plan.md`は以下のような構造を持つ：

```markdown
# 実装計画: [機能名]

## 概要
[機能の説明とビジネス成果]

## アプローチ
[実装方法の詳細な説明]

## 変更されるファイル
- `path/to/file1.ts`
- `path/to/file2.ts`
- ...

## 実装詳細

### 1. [変更1のタイトル]
[説明]

```typescript
// コードスニペット
```

### 2. [変更2のタイトル]
[説明]

```typescript
// コードスニペット
```

## 考慮事項とトレードオフ
- [考慮事項1]
- [考慮事項2]
- ...

## 実装順序
1. [ステップ1]
2. [ステップ2]
3. ...
```
