# commit-message

現在の変更内容から **Conventional Commits 形式・日本語** のコミットメッセージ案を作成し、ユーザーに提示する。**コミットは実行しない**。

## 動作手順

1. **変更を把握する**（並列で実行）
   - `git status` で対象ファイルを確認
   - `git diff --staged`（ステージ済みがあればこれを優先）／なければ `git diff`
   - 必要に応じて `git diff --staged --stat` で全体像を把握
   - ステージ済みも未ステージもない場合はユーザーに確認

2. **既存の慣習を踏襲する**
   - `git log -n 20 --pretty=format:"%s%n%n%b%n---"` で直近のメッセージスタイルを確認
   - そのリポジトリで使われている type（`feat`, `fix`, `chore`, `refactor`, `docs`, `test`, `style`, `perf`, `build`, `ci` など）、scope の付け方、タイトル長、本文の書き方を**そのリポジトリのスタイルに合わせる**
   - 既存メッセージが英語中心であっても、本スキルでは**日本語で出力**する（ただし type 名や scope は既存慣習に従う）

3. **メッセージ案を作成する**
   - **形式**:
     ```
     <type>(<scope>): <タイトル>

     <本文>
     ```
   - **タイトル**:
     - Conventional Commits の type を先頭に付ける
     - scope は変更範囲が明確なときのみ付ける（例: `feat(auth): ...`）
     - 50字程度を目安に簡潔に。末尾に句点は付けない
     - 「何を変えたか」を一行で
   - **本文**:
     - タイトルから1行空けて記述
     - **why（なぜ変えたか）** を中心に書く。what は diff から読めるので、動機・背景・トレードオフを優先
     - 1行72字程度で折り返す
     - 必要なら箇条書きを使う
     - 自明な小変更（typo修正、依存更新のみ等）の場合は本文を省略してよい
   - **破壊的変更**があるときはタイトルに `!` を付ける（例: `feat!: ...`）か、本文に `BREAKING CHANGE:` を含める

4. **提示する**
   - メッセージ案をコードブロックで出力
   - 変更が複数の論理単位に分かれているとき（例: 機能追加 + 無関係な refactor）は、**コミットを分割する提案**を添える
   - 不確実な点（scope の妥当性、type 選定の迷いなど）があれば一言添える

## 出力フォーマット例

````
提案するコミットメッセージ:

```
feat(auth): セッショントークンの保存先を Redis に移行

ローカルメモリ保存ではマルチインスタンス環境でログイン状態が
共有されない問題があったため、Redis ベースのストアに切り替えた。
TTL は既存のセッション有効期限（24h）に合わせている。
```
````

## やらないこと

- `git add` / `git commit` の実行（**提示のみ**）
- ステージされていない変更を勝手にステージすること
- コミットメッセージに `Co-Authored-By` 等のフッターを追加すること（ユーザーが明示的に求めた場合のみ）
- 機密情報（APIキー、パスワード等）が diff に含まれている場合はメッセージを作る前に警告する

## type のガイド（迷ったとき）

| type | 用途 |
| --- | --- |
| `feat` | ユーザーから見える機能追加 |
| `fix` | バグ修正 |
| `refactor` | 振る舞いを変えない内部改善 |
| `perf` | パフォーマンス改善 |
| `docs` | ドキュメントのみ |
| `test` | テストの追加・修正のみ |
| `chore` | ビルド設定、依存更新、雑務 |
| `build` | ビルドシステム・依存関係 |
| `ci` | CI 設定 |
| `style` | フォーマットのみ（ロジック変更なし） |

そのリポジトリで使われていない type は避け、既存ログにあるものを優先する。
