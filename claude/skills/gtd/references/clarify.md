# Clarify（明確化）詳細ワークフロー

Inboxの各タスクに対してClaudeが問いかけ、適切なアクションに変換する。

## 判断フローチャート

```
[タスク名を読み上げる]
     ↓
行動可能？
├─ No → ①参照情報 ②いつかやる ③ゴミ箱 → td delete or @someday
└─ Yes → 2分以内にできる？
          ├─ Yes → 今すぐやって td done
          └─ No → 自分がやる？
                  ├─ No → td edit <id> -l @waiting（委任先を明記）
                  └─ Yes → 複数ステップ必要？
                           ├─ Yes → プロジェクト化 → td edit <id> -p "プロジェクト名"
                           └─ No → Next Actionを設定
                                    → 期日・コンテキスト・優先度を付与
```

## Claudeが問いかける質問

1. 「このタスク、今の形で何をすれば完了しますか？」（曖昧なタスクを具体化）
2. 「誰かを待つ必要がありますか？」（@waitingの判断）
3. 「これ、来週以降でも大丈夫ですか？それとも今週中に必要ですか？」（期日の設定）
4. 「どのプロジェクトに関係しますか？」（プロジェクト割り当て）
5. 「どこでやる作業ですか？（PC前/外出中/電話が必要等）」（コンテキストラベル）

## タスク名の書き換えルール

- 曖昧: 「Aの件」→ 具体化: 「Aに関してBをメールで確認する」
- 名詞: 「プレゼン」→ 動詞: 「プレゼン資料の構成を1ページで書く」
- 大きすぎる: 「プロジェクトX」→ 分割: 最初の次のアクションだけをタスクに

```bash
# タスク名を書き直す
td edit <id> -c "具体的なアクション名"

# プロジェクトへ移動
td edit <id> -p "プロジェクト名"

# ラベル付与
td edit <id> -l @waiting
td edit <id> --add-label @someday

# 期日と優先度
td edit <id> -d "next monday" -P 3

# 不要タスクを削除
td delete <id>
```

## Inbox処理のペース

- 1件あたり30秒〜2分を目安
- 完璧を求めない。「とりあえず正しい場所に移動」でいい
- 迷ったら@somedayへ。Somedayは週次レビューで再判断
